| Module | Role | Size |

## Integration Seams (strongest coupling)

1. **`main.go` → everything** — The central orchestrator wires scrape → storage → promql → rules → web. Blast radius: ~1440 impacted units across 10+ clusters.

2. **`promql/engine.go` ↔ `storage.Queryable`** — The `QueryEngine` interface (line 125) depends on `storage.Queryable`. This is the primary read-path seam. PromQL knows nothing about TSDB — only the `Querier` interface.

3. **`scrape/` ↔ `storage.Appendable`** — The scrape loop ingests samples via `storage.Appendable`. This is the primary write-path seam. Scrape also passes `storage.Storage` for metadata writing.

4. **`storage/remote/` ↔ `tsdb/wal`** — The `QueueManager` is a WAL watcher — it reads the TSDB WAL to ship samples remotely. This is a log-based async coupling.

5. **`rules/` ↔ `promql.QueryEngine` + `storage.Queryable`** — Rules evaluates queries via `QueryEngine` and writes results back through `storage.Appendable`.

6. **`web/` → many** — HTTP API depends on `promql.QueryEngine`, `storage.Storage`, `rules.Manager`, `scrape.Manager`, `notifier.Manager`, `config`, etc.

## Strongest Coupling Points

- **`storage.Storage` interface** (`storage/interface.go:82`) — defines the contract that TSDB implements and that scrape/rules/remote all consume. This is the **keystone interface** of the entire system.
- **TSDB WAL** — used by both TSDB itself and the remote write `QueueManager`. A second, implicit coupling path between TSDB and remote storage.
- **`config.Config`** — consumed by every subsystem. The main `flagConfig` in `main.go:160` is a giant struct with options for all subsystems.
- **`model/` package** — shared types (`labels`, `histogram`, `exemplar`, `metadata`, `relabel`) used across *all* layers.

## Architectural Summary

```
discovery ──► scrape ──► storage.Storage ──► tsdb (local)
                │            │
                │            ├──► storage/remote (WAL watcher → remote write)
                │            │
                ▼            ▼
           rules ──► promql.QueryEngine ──► storage.Queryable
                         │
                         ▼
                      web (HTTP API/UI)
```

The system is a **layered pipeline with a central interface (`storage.Storage`) as the primary decoupling mechanism**. The major integration seams are all defined by Go interfaces in `storage/`, not by concrete types — which is good architecture. The main risk areas are the `config` package (too many consumers with implicit coupling) and `main.go` (too much orchestration logic in one place).
