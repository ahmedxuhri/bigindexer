## Prometheus Architecture Analysis

### **VERIFIED** Architectural Boundaries (Layered Architecture)

The codebase is organized into distinct layers with clear interface-based boundaries:

**Layer 1: Core Domain Models** (`model/`, `prompb/`)
- Labels, histograms, exemplars, metadata, textparse, relabel, value, rulefmt, timestamp
- Protocol buffer definitions
- **No imports from** `storage/`, `scrape/`, `promql/`, `rules/`, `web/`
- Source: `model/labels/`, `model/histogram/` — leaf packages

**Layer 2: Storage Abstraction** (`storage/`)
- `storage.Storage`, `storage.Appendable`, `storage.AppendableV2` interfaces at `storage/interface.go:82-97` and `storage/interface_append.go:27-33`
- `storage.Fanout` multiplexes primary (TSDB) + secondaries (remote) at `storage/fanout.go:45-51`
- Querier/Appender interfaces are the central contract

**Layer 3: TSDB** (`tsdb/`)
- Implements `storage.Storage` — the primary storage backend
- Sub-packages: `chunkenc/`, `chunks/`, `index/`, `wlog/`, `tombstones/`, `record/`, `encoding/`, `compression/`

**Layer 4: Modules (Scrape, Rules, PromQL, Notifier)**
- All depend on `storage.Storage` interface — never directly on TSDB
- Scrape: writes via `storage.Appendable` / `storage.AppendableV2` at `scrape/manager.go:51-58`
- Rules: uses `storage.Appendable` + `storage.Queryable` at `rules/manager.go:34-41`
- PromQL: queries via `storage.Queryable` at `promql/engine.go`

**Layer 5: Web + API** (`web/`)
- Orchestrator handler — depends on **all** modules: `scrape.Manager`, `rules.Manager`, `promql.Engine`, `notifier.Manager`, `storage.Storage` at `web/web.go:224-232`
- Imports: `config`, `notifier`, `promql`, `rules`, `scrape`, `storage`, `web/api/v1`

**Layer 6: Config** (`config/`)
- Global configuration — imported by every module (scrape, discovery, remote storage, web, TSDB, notifier, rules)
- Contains scrape configs, remote write configs, service discovery configs, alerting configs

**Layer 7: Discovery** (`discovery/`)
- Provider pattern — multiple implementations (kubernetes, consul, aws, azure, etc.)
- Communicates via `discovery/targetgroup.Group` channel at `discovery/manager.go`
- Consumed by scrape manager and notifier manager via `SyncCh()`

**Layer 8: Remote Storage** (`storage/remote/`)
- Implements `storage.Storage` — secondary/best-effort backend
- Queue manager, codecs, write/read handlers, OTLP translation

### **VERIFIED** Strongest Coupling Points

**1. `cmd/prometheus/main.go` (The God Constructor) — `main.go:892-1023`**
- **Strongest coupling** in the system. Manually wires every module:
  ```go
  var (
      localStorage  = &readyStorage{...}
      scraper       = &readyScrapeManager{}
      remoteStorage = remote.NewStorage(...)
      fanoutStorage = storage.NewFanout(logger, localStorage, remoteStorage)
  )
  ```
- Injects `fanoutStorage` into `scrape.NewManager()` at `main.go:944`
- Creates `promql.Engine` with `localStorage` as queryable at `main.go:960-978`
- Creates `rules.Manager` with both `fanoutStorage` (appendable) and `localStorage` (queryable) at `main.go:982-1002`
- All 8 reloers listed at `main.go:1056-1166` directly reference concrete module instances

**2. `web/web.go` — `Options` struct at `web/web.go:269-319`**
- Central dependency hub: holds pointers to `*scrape.Manager`, `*rules.Manager`, `*promql.Engine`, `*notifier.Manager`, `storage.Storage`
- Any new module feature exposed via API needs new fields here — creates a bottleneck

**3. `storage/fanout.go` — `fanout` struct at `storage/fanout.go:29-34`**
- The **write path coupling point**: all scraped data flows through `fanoutAppender` → `primary` (TSDB/Agent) + `secondaries` (Remote Storage)
- Primary is authoritative; secondary errors are demoted to warnings
- Read path merges from primary (authoritative) + secondaries (best-effort)

**4. `storage/remote/storage.go` — Remote storage as `secondary`**
- Tightly coupled to TSDB (needs `StartTime()` from local storage) at `cmd/prometheus/main.go:895`
- `SetWriteNotified` call bridges TSDB → remote storage at `main.go:1475`
- Queue manager has complex coupling between write-ahead semantics and remote write protocol

**5. `config/config.go` — Global Configuration**
- Imported by: `scrape`, `storage/remote`, `web`, `rules`, `discovery`, `notifier`, `cmd/prometheus`
- Any config field addition cascades changes across all these modules

**6. Agent Mode — `readyStorage` at `main.go:1761-1816`**
- Runtime type-switch on `*tsdb.DB` vs `*agent.DB` across 7+ methods (`Querier`, `Appender`, `ExemplarQuerier`, `CleanTombstones`, `BlockMetas`, `Delete`, `Snapshot`, `Stats`)
- **HYPOTHESIS**: This pattern suggests incomplete abstraction — if `agent.DB` and `tsdb.DB` shared a richer interface, the type-switches would disappear

**7. Rules → PromQL — `EngineQueryFunc` at `rules/manager.go:50-73`**
- Rules manager takes a `QueryFunc` closure that captures `promql.Engine` + `storage.Queryable`
- Can't create rules without a query engine; can't evaluate rules without both

### Summary

| Perspective | Key Insight |
|---|---|
| **Strongest boundary** | `storage.Storage` interface separates TSDB from all consumers (scrape, promql, rules) |
| **Primary integration seam** | `storage.Fanout` — the write/read multiplexer between TSDB and remote storage |
| **Tightest coupling** | `cmd/prometheus/main.go` — the composition root that wires 8+ modules manually |
| **Most central module** | `web.Handler` — depends on all modules; every feature adds a field to its `Options` |
| **Hidden coupling** | Agent mode type-switches on `readyStorage` — TSDB and Agent should share more interface surface |
