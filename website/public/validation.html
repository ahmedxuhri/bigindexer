Notable Findings

pydantic-core — biggest win

Baseline p01: evidence coverage 0%, boundary accuracy 0 — the model had no idea about the Rust/Python split architecture.
MCP p01: evidence coverage 80%, boundary accuracy 1 — BGI injected the exact module boundaries, the model got it right first try.

django — consistent lift

Evidence coverage increased from 73.3% → 84.0% (+10.7%). Django's explicit app structure helps baseline too, but MCP delivered higher coverage across all 4 prompts. Latency dropped 26% (99.8s → 73.1s) because pre-indexed context shortens the exploration phase.

fastapi — what the evidence actually shows

Honest finding: Evidence coverage dropped on two fastapi prompts with MCP (p03: 33.3%, p04: 66.7% vs baseline 90%, 100%). This is real, and it deserves a real explanation — not "prompt-tuning issue."

Reading the raw outputs reveals the cause: the baseline model, having no architecture context, was forced to read every source file manually and produced a 10-item verified claim table. The MCP model received blast-radius data (1,614 impacted units, top seam confidence 0.75) and accepted that as the architecture picture — making fewer granular file-level verifications.

What this means: MCP trades file-reading depth for architectural accuracy. On well-structured repos, baseline file-reading can score high on a granular checklist. MCP wins most where the baseline is architecturally blind (pydantic: 0% → 80%). Boundary accuracy was perfect in both fastapi modes — MCP understood the architecture correctly, it just verified fewer granular claims explicitly.

The raw outputs for all fastapi runs are in validation/runs/fastapi/ on GitHub. Read them and judge for yourself.

prometheus (Go) — cross-language reality check

Prometheus expanded this validation beyond Python. After post-routing retest rows were added, evidence coverage is 90.0% in both modes and boundary accuracy remains perfect (1.0). MCP is still slower on median latency (119.9s vs 89.9s), which remains important to report publicly. In the post-routing first pass, MCP invocation was confirmed on p02/p03 while p01/p04 remained invalid/unscored. This indicates MCP gains are strongest when baseline models are architecturally blind (as in pydantic-core), and less dramatic when baseline exploration is already strong.

next.js (TypeScript) — large monorepo signal

Next.js adds a large TypeScript monorepo to the dataset. In the scored prompts, MCP slightly improved evidence coverage (89.2% → 91.7%) with perfect boundary accuracy in both modes, while reducing median latency from 291.8s → 66.4s. One MCP p04 run did not invoke MCP tools and is marked invalid/unscored, which is why Next.js currently shows 3 scored prompt pairs.

Hallucination rate: 0 in both modes

No factually incorrect module or file claims were observed in any of the 40 scored runs. MCP context grounds the model rather than introducing new errors.

Limitations — Read This

We believe in publishing limitations alongside results. A reader who finds these themselves will trust the evidence less than a reader we told first.

Self-reported scoring

The ground-truth checklists were written by us, and the runs were scored by us. The checklists were defined before scoring by reading the actual source code, and the full scoring rubric is published at validation/SCORING_RUBRIC.md. Every raw AI output is committed to GitHub. Anyone can re-score independently — and we explicitly invite it. If you score any run differently, open an issue with your reasoning.

Sample size is still limited

40 scored runs across 5 repos (Python + Go + TypeScript) is stronger than the initial batch, but still not conclusive. The pydantic-core finding is strong enough to stand on its own, but the aggregate story still needs more language diversity (especially large Java/C# monorepos and independent replications).

Actionability was flat

Actionability scored 4.0–4.3/5 in both modes across all repos. MCP did not improve actionability scores. This is honest signal: MCP helps the model understand the architecture, but does not yet produce more actionable output. Combining MCP context with more directive prompts ("use MCP AND provide copy-paste ready code") is an open improvement area.

Shipped after this validation batch (2026-05-11): BGI-TWIN MCP context package: task_fingerprint, behavioral_twins, and twin_context (top-3 twins + seam + rubric + confidence gate). Public metrics above are pre-refresh; rerun is next.

We need one independent replication

The single thing that would most improve this evidence: one engineer who has never used Big Indexer running these prompts on their own repo and publishing their raw results. If you're willing to do this, open an issue and we'll send you the exact protocol.

Reproduce This

Every run is reproducible. Clone the repo, scan, start the MCP server, run the same prompts.

# Install
pip install bigindexer

# Clone any target repo (shallow)
git clone --depth 1 https://github.com/tiangolo/fastapi

# Scan → produces bgi-graph.json + fuse-graph.json
bgi scan fastapi/ --out output/

# Start MCP context server
bgi mcp --graph output/bgi-graph.json --fuse-graph output/fuse-graph.json

# Configure opencode (opencode.json in target repo):
# { "mcp": { "bgi": { "command": "bgi", "args": ["mcp", ...] } } }
opencode  # BGI context auto-injected via MCP

Full run logs, scores, and the complete scoring rubric are in validation/ on GitHub . Setup guide in docs/MCP_SETUP.md .
